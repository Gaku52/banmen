# Banmen Operating Plan v0.1

## 1. Purpose

Banmen を「思考停止を減らす道具」として成立させるための、実行・計測・収益化の運用設計を定義する。

---

## 2. KPI Definitions

## 2.1 Core KPI

- `A1` (Activation 1): 初回起動から30秒以内に1件追加したユーザー率
- `D7` (Day 7 Retention): 初回利用から7日後に再訪したユーザー率
- `W3` (Weekly 3): 週3回以上、1分以内セッションを行ったユーザー率
- `R1` (Relief 1): 週1の自己申告で「なんだっけが減った」と回答した率

## 2.2 Target (First 90 days)

- A1: `>= 65%`
- D7: `>= 30%`
- W3: `>= 20%`
- R1: `>= 60%`

---

## 3. Event Tracking Spec

最低限、以下のみを計測する。

- `app_open`
- `task_added`
- `task_promoted_to_now`
- `now_completed`
- `undo_used`
- `micro_prompt_shown`
- `micro_prompt_submitted`
- `signup_started`
- `signup_completed`
- `sync_enabled`
- `paywall_viewed`
- `checkout_started`
- `checkout_completed`

ルール:
- 個人タスク本文は収集しない
- User ID は匿名IDを基本（任意登録時に連携）
- プライバシーポリシーにイベント収集を明示

---

## 4. Micro Prompt Design

## 4.1 Initial 9 Prompts

着手系:
- 「次の10分で終わる1つは？」
- 「今すぐ手をつけるなら何？」
- 「先延ばししている1つは？」

優先順位系:
- 「今やる価値が一番高いのは？」
- 「終えると一番軽くなるのは？」
- 「今日これだけ進めるなら何？」

詰まり解除系:
- 「迷っている理由を1行で書くと？」
- 「最初の1手だけ書くなら？」
- 「忘れる前に置くべき1つは？」

## 4.2 Display Rules

- NOW が空のときのみ表示
- 1起動で1問のみ
- 同一問いの再表示は24時間空ける
- スキップ可能（強制なし）

---

## 5. Monetization Spec

## 5.1 Pricing Hypothesis

- Free: コア機能（NOW1/NEXT3、3秒Undo、ローカル保存）
- Pro (年額中心): `現盤面同期` + `端末紛失時復元` + `優先サポート`

暫定価格案:
- 月額: 300円〜500円
- 年額: 2,400円〜4,000円

## 5.2 Paywall Trigger

- 2台目接続時に表示
- 「同期は任意。ローカル利用は無料で継続可」を明記

## 5.3 Success Metrics

- 無料 -> 有料転換率: `>= 3%`
- 有料月次解約率: `< 5%`
- LTV/CAC: `> 1.5` を継続

---

## 6. Sync Spec (Future-ready)

## 6.1 Sync Scope

同期するのは「現在盤面のみ」:
- nowTask
- nextTasks (max 3)
- updatedAt

同期しない:
- 完了履歴
- 統計
- 過去ログ

## 6.2 Conflict Resolution

- 基本: last-write-wins
- NOW編集中は3秒の軽い編集ロック
- 衝突時は小さな通知のみ表示（復元履歴は持たない）

## 6.3 Data Expiry

- 長期未使用データは自動失効（例: 7日〜30日）
- 失効期間は検証で調整

## 7. 30/60/90 Day Execution Plan

## 7.1 Day 0-30

- 1画面PWAのMVP公開
- A1/D7計測導入
- 10人ユーザーテスト実施
- マイクロ問い9種の初期検証

## 7.2 Day 31-60

- 任意登録導線（ログインなし継続利用可）
- 同期価値のモック検証（有料意向調査）
- 価格帯テスト（2案比較）

## 7.3 Day 61-90

- Pro最小版リリース（現盤面同期）
- X / Product Hunt / Show HN / Reddit で配信
- 有料転換率・解約率の最初の実測

---

## 8. Go-to-Market Posting Cadence

- X: 週5投稿（開発ログ、10秒デモ、ユーザー学び）
- Product Hunt: ローンチ当日集中対応（コメント返信重視）
- Show HN: 実装詳細と学びを透明に共有
- Reddit: ルール順守でコミュニティごとに文脈最適化

---

## 9. Non-Negotiables

- 1画面原則を崩さない
- NOW1/NEXT3制約を崩さない
- 履歴を価値化しない
- 強制通知で継続を作らない
- プライバシー最小収集を維持

---

## 10. Weekly Review Template

- 今週のA1 / D7 / W3 / R1
- もっとも効いた改善1つ
- 削除した機能・文言1つ
- 来週の実験仮説1つ

---

## 11. Spec Lock (再相談を減らすための固定仕様)

## 11.1 Priority Order

判断が割れた場合は、以下を上から優先する。

1. 認知摩擦の削減
2. 入力速度
3. シンプルさ
4. 収益性
5. 見た目

## 11.2 Default Decisions

未定義ケースは以下をデフォルト採用する。

- タスク本文の上限: 80文字
- 空文字/空白のみ: 保存不可
- NEXTが3件のとき: 追加不可、ユーザーに短い理由表示
- NOWが埋まっているときの昇格: 入れ替え確認を出す
- 完了操作: 即非表示 + 3秒Undo
- Undoしなかった場合: 完全削除（履歴なし）
- オフライン時: ローカル保存優先、同期は復帰時に再送
- 同期衝突: last-write-wins
- 重大障害時: 同期を一時停止しローカル利用を維持

## 11.3 Copy Lock

以下の文言は原則固定する。

- 「今やることを1つ、次を3つまで置けます。」
- 「終わったら消します。履歴は残しません。」
- 「迷ったら、まず1行だけ書いてください。」

変更条件:
- A/Bテストで指標改善が有意な場合のみ変更

## 11.4 Scope Guard

以下の要求は原則Reject（別Phaseに送る）。

- タグ、カテゴリ、プロジェクト
- 期限、通知、カレンダー連携
- 完了履歴、統計ダッシュボード
- 長文ノート、添付ファイル

## 11.5 Release Gate

リリース前に以下を満たさない場合は出さない。

1. 起動から10秒以内に1件追加できる
2. NOW1/NEXT3制約が壊れない
3. 完了 -> 3秒Undo -> 完全削除が動作する
4. リロード後に盤面が復元される
5. 主要導線でクラッシュしない

## 11.6 Change Policy

- 仕様変更提案は毎週1回だけ受付
- 同時に変更できる主機能は1つまで
- 新機能追加時は「削除する要素」を1つセットで提出

## 11.7 Unknowns Backlog

未確定事項はこの形式で記録する。

- `ID`
- `論点`
- `暫定デフォルト`
- `検証方法`
- `期限`

このフォーマット以外の未確定要素は、実装を止めず暫定デフォルトで進める。
